Version 5.0.0 - The "Game Library Update" (21/03/2026)
------------------------------------------

✨ NEW FEATURES:
• New FoggyBytes icon: StreamTweak now sports a new icon that visually unifies the FoggyBytes suite — the same icon is shared with StreamLight for a consistent identity across both apps
• Game Library Sync: StreamTweak automatically discovers installed games across Steam, Epic Games, GOG, Ubisoft Connect, Xbox App, EA App, and Battle.net — and syncs them to Sunshine (and compatible forks: Apollo, Vibeshine, Vibepollo) without touching manually created entries; uninstalled games are removed on the next sync
• Cover art from native sources: cover images are fetched from each store's own CDN (Steam CDN, Epic metadata, GOG API, Battle.net aggregate.json) and cached locally as PNG — Sunshine displays them natively; no third-party service involved
• Battle.net integration: Battle.net games are detected via the local aggregate.json file; all entries launch via the Battle.net client (Battle.net.exe), so any installed game is automatically included without needing per-game exe paths
• Manual game management: use the Add button in the Games tab to include any executable not auto-detected; manual entries survive re-syncs and are removed individually with the − button per row
• APPSTORES TCP command: StreamTweak serves the game→store map to StreamLight via the new APPSTORES command on port 47998, enabling per-game store badge display in the StreamLight client

🔧 IMPROVEMENTS:
• Game list sorted alphabetically after each sync for a consistent, predictable order
• Manual games persist across re-syncs: only auto-discovered games are refreshed; manually added entries are preserved until explicitly removed
• Single-entry Sunshine sync: adding or removing a manual game updates only that entry in apps.json — no other entries are touched; compatible with apps created outside StreamTweak

⚠️ REQUIRES: StreamLight 2.0.0 or later for store badge display in the client.

Version 4.4.0 - The "Telemetry Update" (20/03/2026)
------------------------------------------

✨ NEW FEATURES:
• Host metrics collection: StreamTweak now collects real-time host PC metrics — GPU usage %, GPU encoder usage %, GPU temperature (°C, NVIDIA only via NVML), VRAM used (MB), VRAM total (MB, NVIDIA only), CPU usage %, and network TX (Mbps) — and serves them to StreamLight via a new STATS command on the TCP bridge (port 47998)
• STATS command: the TCP bridge accepts a new STATS command that returns a JSON payload (e.g. {"gpu":45,"gpu_enc":12,"gpu_temp":62,"vram_used":4096,"vram_total":8192,"cpu":18,"net_tx":80}) for consumption by StreamLight's overlay
• Cross-vendor GPU metrics via PDH PerformanceCounters: GPU utilization (3D engine), GPU encoder utilization, and VRAM used are read with System.Diagnostics.PerformanceCounter using the GPU Engine and GPU Process Memory performance counters — works on NVIDIA, AMD, and Intel Arc
• NVML P/Invoke for GPU temperature and total VRAM (NVIDIA only): temperature and total VRAM capacity are read directly from nvml.dll (shipped with NVIDIA drivers) via P/Invoke; on non-NVIDIA systems these fields report -1 and no error is raised
• Metrics unavailable for StreamLight: if StreamTweak is not reachable or metrics are not available, StreamLight's overlay omits the host metrics section entirely — no fallback values, no placeholders

🔧 IMPROVEMENTS:
• HostMetricsCollector initialises asynchronously on a background Task to avoid any impact on WPF startup time
• GPU Engine PDH counter list is rebuilt every 60 seconds to handle processes appearing and disappearing at runtime (instances are per-process and ephemeral)
• VRAM reads use RawValue (int64) instead of NextValue (float) to avoid precision loss on systems with more than 2 GB of dedicated VRAM

⚠️ REQUIRES: StreamLight 1.2.0 or later for the host metrics overlay feature.

Version 4.3.0 - The "App Manager Update" (19/03/2026)
------------------------------------------

✨ NEW FEATURES:
• Streaming App Manager: a user-configurable list of executables that StreamTweak automatically terminates when a streaming session starts and relaunches when it ends — ideal for apps like Hue Sync that conflict when running on both host and client simultaneously; each entry has an independent AutoManage toggle so individual apps can be excluded from automation while staying in the list
• App Manager manual controls: each listed app can also be killed or restarted on demand from the Apps tab, without waiting for a streaming session to trigger the automation

🔧 IMPROVEMENTS:
• Session logging now tracks all streaming sessions regardless of whether NIC throttle was applied: previously only sessions that triggered a speed change were logged; now every session is recorded with a NIC Throttle column (Yes/No) and an Original NIC Speed column; the old Mode column has been removed
• Session date format now includes the year (dd/MM/yyyy) for unambiguous historical records
• Dolby Atmos retry loop: after the initial 30-second wait, DolbyAudioMonitor now retries device discovery every 5 seconds for up to 3 minutes — handles the case where the Windows Audio service is still initializing when StreamTweak starts on an early auto-login

Version 4.2.4 - The "Under the Hood Update" (18/03/2026)
------------------------------------------

🐛 STABILITY & FIXES:
• Fixed AutoHdrToggle_Changed reverting with the wrong guard flag on error: _hdrToggleBusy was set/cleared instead of _autoHdrBusy, which silently suppressed all subsequent Auto HDR toggle events and could incorrectly block HDR monitor toggles
• Fixed ApplySpeedViaPowerShell always returning true regardless of the PowerShell process exit code: failed speed-change commands were silently reported as successful to the pipe client

🔧 IMPROVEMENTS:
• Removed dead null-checks on GetSupportedSpeeds return value: the method always returns a (possibly empty) Dictionary — never null; all five redundant call-site guards in App.xaml.cs have been removed
• Removed superfluous CreateNoWindow = true from ApplyWithUac in SpeedChanger: the property is silently ignored by Windows when UseShellExecute = true
• Extracted shared DebugLogger class: the identical DebugLog implementation previously duplicated across App, StreamTweakBridge, LogParser, and StreamingLogMonitor is now consolidated in a single StreamTweak.DebugLogger.Log entry point
• Centralized config persistence with PatchConfig helper: the repeated JSON read/patch/write pattern previously spread across seven SaveXxx methods in App.xaml.cs and SettingsWindow.xaml.cs is now unified behind a single PatchConfig(Action<Dictionary>) helper in each class

Version 4.2.3 - The "Display & Logs Refinement Update" (16/03/2026)
------------------------------------------

🔧 IMPROVEMENTS:
• Display tab — simplified context label: removed ambiguous "Physical" / "Virtual" detection logic and replaced it with a straightforward display of the detected streaming app name (Apollo, Vibeshine, Vibepollo, or Sunshine) — the tab shows all active monitors regardless, keeping the interface simple and reliable
• Logs tab — lightning icon refinement: the ⚡ symbol now appears only for truly interrupted sessions (app crash or forced restart), not for normal stream disconnects; a session is marked interrupted only if it was never properly closed (`EndReason == "Interrupted"`); this eliminates false positives where normal closes were misidentified
• Logs tab — footer removed: removed the confusing footer text explaining the lightning icon, as the new behavior is self-evident; cleared additional space for the session list to breathe
• Session logger — race condition fix: `SessionLogger.EndSession()` now only sets `EndTime` and `EndReason` once per session, preventing duplicate close attempts from the log monitor and bridge handler from interfering with each other
• Log monitor — state machine: added a simple state tracker (`seenStreamStarted`) to `StreamingLogMonitor` that prevents spurious `StreamStopped` events (from log rotation, stray lines, or dynamic logs appearing late) from triggering false session closes if no `StreamStarted` was seen first

🐛 STABILITY & FIXES:
• Fixed lightning icon appearing incorrectly for normally closed streaming sessions
• Fixed occasional double-close of sessions when both the log monitor and TCP bridge tried to end the same session

Version 4.2.2 - The "Bridge Reliability Update" (15/03/2026)
------------------------------------------

🐛 STABILITY & FIXES:
• Fixed PREPARE command from StreamLight being silently ignored when Auto Streaming Mode was disabled: the TCP bridge handler now works independently from the Auto Mode toggle — PREPARE correctly triggers the NIC speed change regardless of whether Auto Mode is on or off
• Fixed UI thread blocking during PREPARE handling: the handler was using Dispatcher.Invoke with an async lambda, which resolved to async void and ran the named-pipe call to StreamTweakService synchronously on the UI thread (up to 5 seconds freeze); switched to Dispatcher.InvokeAsync and Task.Run so the blocking pipe roundtrip no longer stalls the interface

Version 4.2.1 - The "Reliability Update" (15/03/2026)
------------------------------------------

🐛 STABILITY & FIXES:
• Fixed Auto Streaming Mode toggle being ignored on new client connections: when the toggle was disabled, the TCP bridge handler (PREPARE command from the StreamLight client) still triggered a NIC speed change because it did not check the Auto Mode state; the speed change is now fully blocked when Auto Mode is off
• Fixed Auto Streaming Mode toggle being ignored via log-monitor path: HandleAutoStreamStart() also lacked the Auto Mode guard, meaning a stream-start log event could still change the NIC speed — particularly when Dolby monitoring was active and kept the log monitor running; both entry points now respect the toggle
• Fixed race condition in auto stream start: if the user disabled Auto Mode during the 7.9-second countdown alert, the speed change was still applied after the delay; the toggle state is now re-checked immediately after the countdown before any speed command is sent

🔧 IMPROVEMENTS:
• Display tab: removed placeholder cards for disconnected monitors (Display 2, Display 3); the panel now only shows monitors that are actually connected, keeping the Display section clean on single-monitor setups
